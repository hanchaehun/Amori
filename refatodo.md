# AMORI — 페르소나 충실도 개선 계획 (2026-07-15 확정)

> 목표: 사용자가 미리보기를 본 순간 "어, 나네"라고 느끼는 페르소나.
> **설계·문항 구성의 심리학적 근거와 출처는 `docs/persona_science_rationale.md`** —
> "왜 이렇게 했더라"는 그 문서에서 찾는다. 기술 설계는 `docs/persona_fidelity_design.md`.
> **보류 결정(7/15)**: 대화형 온보딩 전환은 팀 합의 대기, 카톡 import는 진행하지 않음.
> 이 문서는 그 둘을 제외한 실행 설계다.

## 진단 요약 — 왜 지금은 "나 같지" 않은가

1. **바넘 효과**: 객관식 5답으로 trait 8개를 강제 생성 → 최소 3개는 LLM의 지어낸 문장.
   누구에게나 맞는 말은 "내가 아닌" 느낌의 주범 (Stanford 1,052명 연구: 자유 응답 인터뷰
   대비 설문+인구통계 기반 재현율이 유의미하게 낮음).
2. **수정권 부재**: 틀린 추측을 사용자가 고칠 방법이 없어 그대로 소개팅을 다닌다.
3. **심리 기저층 미구현**: 존댓말·ㅋㅋ가 같아도 "서운할 때 반응"이 다르면 남이다.
   설계 §9의 2층(화용 행동)·3층(심리 기저)이 계획만 있고 시뮬에 안 들어간다.
4. **표본 부족의 상한**: few-shot 2~3개 ICL은 평균 스타일로 회귀(Catch Me If You Can '25).
   카톡 import 보류이므로 v1 해법은 "미리보기 수정문 수집 + 데일리 뱅크"다.

## P0-A. trait 가변화 — 바넘 제거

근거 있는 카테고리만 생성하고, 없는 것은 "아직 몰라요"로 정직하게 보여준다.

- **스키마**: `PersonaTrait` += `confidence: float(0~1)`, `evidence: list[str]`(근거 답변 코드,
  예 `["R-2:A"]`). `PersonaOutput`/`PersonaResponse`의 traits `min_length=8` → `min 1, max 8`.
  카테고리 enum(8종)은 유지 — 순서 고정은 해제.
- **프롬프트**(`prompts/persona.py`): "8개 채우기" 지시 삭제 → "답변이 근거를 주는 카테고리만
  생성하라. 각 trait에 근거 답변 코드를 evidence로 명시하라. 근거 없는 카테고리는 만들지 마라."
- **modoo 주의**: 스키마 강제가 안 되는 provider — Pydantic 검증 실패 시 1회 재시도 경로 확인.
- **Flutter**: 페르소나 화면에서 빈 카테고리는 "아직 알아가는 중이에요 — 내일 질문에
  답하면 채워져요" 카드 + 데일리 CTA. (빈 칸이 오히려 데일리 리텐션 훅이 된다.)
- **임베딩**: `persona_embedding_text`는 존재하는 trait만 순회 — 기존 로직 그대로 호환.
- **하위호환**: 기존 8-trait 행은 그대로 유효(max 8). confidence/evidence 없는 기존 trait은
  기본값(0.5, [])로 읽는다.

## P0-B. 심리 기저층 — psych_profile + conversation_policy (설계 §9-4 구현)

"서운할 때 어떻게 구는지"를 페르소나에 넣는다. **애착 축이 데이팅 도메인 최강 예측
변수**(ECR 계열, PRISMA 리뷰 검증)이고, R-2/R-3/R-4가 이미 사실상 애착 문항이다 —
새 설문 없이 매핑만 한다.

### 데이터 (personas에 JSONB 2컬럼 추가, alembic 신규 리비전)

```jsonc
"psych_profile": {
  "big_five": {"E": 0.3, "A": 0.7, "C": 0.6, "N": 0.4, "O": 0.5,
               "evidence": ["R-2:A", "mbti_prior"], "confidence": 0.4},
  "attachment_hint": "안정-약간 불안",   // hint 어투만 — 단정 금지
  "user_visible": true                  // 미리보기에서 공개·수정 가능 (프라이버시 원칙)
},
"conversation_policy": {
  "question_ratio": 0.35,           // voice_stats에서 복사 (실측)
  "reaction_amplitude": "큼",        // voice_stats 감탄사·!!·이모지 밀도에서 코드 산출
  "conflict_mode": "지연-정리형",     // R-3 결정적 매핑: 즉시형|지연-정리형|회피형
  "reassurance_seeking": "낮음",     // R-2 결정적 매핑: 낮음|중간|높음
  "self_disclosure_pace": "보통"     // R-1·관계 속도 답변 매핑, 근거 없으면 "보통"
}
```

### 산출 규칙 — LLM 추측 최소화 원칙 유지

- **결정적 매핑(코드)**: conflict_mode·reassurance_seeking·self_disclosure_pace는
  객관식 답변→값 고정 테이블(`app/llm/psych_mapping.py` 신설). question_ratio·
  reaction_amplitude는 voice_stats에서 계산. **LLM이 아니라 코드가 만든다.**
- **LLM 추정**: big_five만 — 페르소나 생성 프롬프트에 5축 추정 + evidence 지시 추가.
  MBTI prior(P0-E)가 있으면 초기값으로 합성(증거 수 가중 평균 — 실답변이 쌓이면
  prior 영향 자연 감소).
- **애착 매핑**: R-2(답장 뜸→불안 신호), R-3(서운함 대처→접근/회피), R-4(위로 수용
  방식) 조합 → hint 문구 사전(코드). ECR 문항을 새로 받지 않는다(설문 피로 금지).

### 소비처

- **시뮬 프롬프트**(`_speech_block` v3): 측정값 카드(1층, 기존) 아래 행동 지시(2층) 추가 —
  "3번에 1번 되묻기 | 리액션 큼 | 갈등 시 바로 말하기보다 정리 후 이야기 | 자기 얘기는
  천천히". 3층은 트레이트 마커 지시 1줄("부정 정서 단어 드묾, 완화 표현 자주" 식).
- **조율 허용 범위 명시**(근거: rationale §7 CAT/LSM): "화제·에너지는 상대에게 수렴해도 된다. 어미·습관·이모지
  ·부호는 절대 고정" — 현행 전면 금지의 부자연스러움 해소.
- **리포트**: attachment_hint는 내부 신호로만 — 리포트 문구에 성격 단정 표현 금지.
- **미리보기**: psych_profile은 사용자에게 공개하고 수정·숨김 가능(P0-C와 결합).

## P0-C. 페르소나 미리보기·수정 (구 P0-5, 범위 확장) — ★ 체감 레버리지 1위

생성 직후 "당신의 에이전트는 이렇게 말해요"를 보여주고, 모든 층을 사용자가 고칠 수 있게 한다.
수정문 = `source=user_written` 최고 등급 데이터가 공짜로 쌓이는 루프.

### BFF

- **`POST /persona/preview`** (신설): 상황 3개(첫인사 / 공감 리액션 / 완곡 거절)에 대한
  에이전트 발화를 생성해 반환. 스타일 게이트 통과 후 반환(시뮬과 동일 경로).
  응답: 발화 3개(+상황 라벨) + trait 카드 + 말투 요약(voice_stats 사람 읽기 문구)
  + psych_profile(hint 어투).
- **`PATCH /persona/me`** (신설): 부분 수정 —
  - `trait_edits`: summary 수정 / 삭제("이건 나 아니에요") → 해당 trait에
    `user_edited: true` 마킹. **이후 데일리 update 병합 시 코드에서 보호**(LLM 출력이
    user_edited trait을 덮지 못하게 병합 로직에서 차단 — 프롬프트 의존 금지).
  - `utterance_fixes`: 미리보기 발화 수정문 `[{register, text}]` → sample_bank 추가
    (source=user_written) + voice_stats 재계산 + sample_messages 갱신.
  - `speech_edits`: verbal_habits·punctuation_habits 직접 편집 + **자유 입력 필드
    "평소에 이렇게 써요"** — 의도적 오타·습관 감탄사·줄임말을 그대로 쓰도록 유도
    (UI hint: "오타도 습관이면 그대로! 예: 마쟈마쟈, 넹넹, 구캥"). 입력문은
    sample_bank(register="자유입력")로 저장되어 말투 통계·few-shot에 반영.
  - `psych_edits`: attachment_hint 숨김, big_five 축 조정.
  - 공통 후처리: traits/values 변경 시 임베딩 재계산, persona_revision 증가.

### Flutter (신규 화면 1 + 진입점 2)

- 페르소나 생성 완료 직후 자동 진입 + 페르소나 탭에서 "미리보기·수정" 상시 진입.
- 발화 카드 3개: 탭하면 인라인 편집("내 말투로 고쳐주세요") → PATCH.
- trait 카드: 수정/삭제 스와이프. 빈 카테고리는 P0-A의 "아직 몰라요" 카드.
- 말투 카드: 측정 요약(존댓말 90%·ㅋㅋ 3연속·이모지 안 씀 등) + 습관 편집 + 자유 입력.
- 심리 카드: hint 어투로 표시("당신은 답장이 뜸하면 신경 쓰이는 편 같아요"), 수정/숨김.

## P0-D. 의도적 오타·표기 보존 (v1 누수 2곳 수정)

원문 보존 경로는 이미 있으나(주관식 → sample_bank 원문), 프롬프트 2곳이 교정을 유발한다.

- `prompts/persona.py`: "사용자가 쓴 문장을 **명백한 오타만 다듬어** 그대로 쓰세요" →
  "**한 글자도 다듬지 말고** 그대로 쓰세요. 오타·줄임·비표준 표기도 그 사람의 목소리다."
- `prompts/simulation.py` `_speech_block`: 발화 예시 아래 지시 추가 — "예시의 맞춤법·표기를
  교정하지 마라. 비표준 표기(의도적 오타·줄임)가 있으면 같은 방식으로 재현하라."
- 스타일 게이트(결정적 후편집)와 충돌 없음 — 게이트는 미보유 습관 제거만 한다.

## P0-E. MBTI 입력 — 프로필 가치 + 약한 prior

받는 이유는 과학이 아니라 제품(전 국민이 아는 공짜 데이터 + 프로필 기대 기능).

- **DB**: `users.mbti` String(4) nullable + alembic 신규 리비전. PUT/GET `/users/me`.
- **Flutter**: 프로필 화면에 16유형 선택 시트(선택 사항), 매칭/리포트 프로필 카드에 표시.
- **prior 변환**(`psych_mapping.py`): E/I→외향성, S/N→개방성, T/F→친화성, J/P→성실성.
  축당 ±0.15, confidence 0.2 고정 — big_five 합성 시 실답변 증거에 밀리는 약한 초기값.
- **금지선(코드 리뷰 체크리스트에 올릴 것)**:
  1. 매칭 점수·랭킹에 MBTI 직접 사용 금지 (유형 궁합표는 유사과학 + Joel 2017).
  2. 시뮬 프롬프트 주입 금지(v1) — 프로필 표시 전용.
  3. "MBTI로 매칭해드려요" 류 문구 금지.

## P0-F. 문항 체계 개편 — 관점 분리 + 온보딩 재구성 (2026-07-15 문항 전수 검토)

32문항 검토 결과: 뼈대(8카테고리×3 데일리 뱅크)는 유지, 구조 결함 3개를 수술한다.
원칙: 축당 2문항이 신뢰도 하한(TIPI/BFI-10, ECR 단축형의 경험칙) — 문항 수가 아니라
축 커버리지가 정확도를 결정한다.

### F-1. 측정 관점 태그 (구조 결함 수정)

- 문항이 **behavior**(나는 어떻게 행동하나 — R-3, 3-x, 5-1, 8-3 등)와 **preference**
  (상대가 그러면 어떻게 느끼나 — 2-x 전부, R-4, 8-1, 5-2 등)로 뒤섞여 있는데
  현재는 전부 trait 생성에 들어간다 → preference 답변으로 만든 trait은 추측이다.
- `Scenario`에 `measures: behavior|preference` 태그 추가(Flutter). **라우팅은 백엔드**:
  behavior → trait evidence + conversation_policy, preference → 정답지/리포트 반응성 축.
- 선택지→심리 축 매핑은 `psych_mapping.py`에 (문항코드, letter) 기준 테이블로 —
  조건부/중립 선택지("상황에 따라 다르다" 류)는 no-signal 처리.
- **humor_style은 객관식으로 만들지 않는다** — 2-x는 전부 상대 유머 수용도(preference).
  내 유머는 주관식(9-2·9-3)과 10-9(농담 응수)의 행동 표본에서만. 초기엔 P0-A의
  "아직 몰라요" 카드로.

### F-2. 온보딩 재구성 (7문항 유지, 구성 교체)

`kInitialScenarioCodes` 변경: `[R-1, R-2, R-3, R-4, R-5, 9-1, 9-2]` →
`[R-1, R-2, R-3, 8-3, R-5, 9-1, 9-2]`

- **R-4(받고 싶은 위로) 제외** — preference 문항이라 trait 근거가 안 됨.
  데일리 첫 주에 정답지 페어("당신은 뭐라고 답하나 + 어떤 답장을 받고 싶나")로 이동.
- **8-3(답장이 짧아졌을 때) 승격** — 뱅크 최고의 애착-불안 문항. R-2와 함께 애착 축
  2문항 하한 충족.
- 여유가 생기면 **9-3(칭찬 반응) 추가로 8문항** — 주관식 3개 = voice_confidence 0.35
  (격식·웃음 습관 안정선). 객관식 추가보다 정확도 기여 큼.
- **R-2 선택지 재작성 제안(문항 텍스트는 팀·명세현 결정)**: 현행 C·D는 불안 축에서
  무신호 — "답장이 뜸하면 신경 쓰여서 자꾸 확인하게 된다" 같은 고불안 선택지 필요.

### F-3. 데일리 뱅크·큐 정비 (감량 아님 — 뱅크는 리텐션 재료)

- 초기-데일리 중복 3쌍 정리: R-2≈1-1, R-3≈3-1, R-4≈8-1 — answered_codes 기준
  큐에서 스킵(문항 삭제 아님 — 온보딩 미응답자는 데일리에서 만나야 함).
- **큐 재배열: "축당 2문항 최속 달성"** — 카테고리 순서 순회 대신, 근거가 부족한
  축(특히 온보딩이 못 잡는 경계선·돈·데이트)을 첫 2주 안에 2문항씩 채우는 순서로.
- P1 10-x 편입 시 객관식:주관식 = 2:1 인터리브(기존 설계 유지).

## P1 (P0 안정화 후)

- **데일리 주관식 뱅크 10-x 12문항 + 큐 2:1 인터리브** (설계 §3-2 그대로) — few-shot 회귀
  문제의 v1 해법. 레지스터 태그가 시뮬 국면별 few-shot 선별(§5-2)의 전제.
- **정답지 페어 수집**: 데일리 주관식과 한 화면 페어("당신은 뭐라고 답하나요" +
  "당신은 어떤 답장을 받고 싶나요") — 백엔드는 수신 준비 완료, 문항·UI만.
- **LLM 저지 충실도 측정**(§6): 실발화 1 + 에이전트 발화 1 블라인드 — 정답률 60% 이하 목표.
  미리보기 수정률(발화 3개 중 몇 개를 고치는가)도 충실도 프록시로 로깅.
- **변형 표기 빈도 측정** (2026-07-15 사용자 지적): 의도적 오타는 "항상 쓰는 것"과
  "가끔 쓰는 것"이 다르다. v1은 프롬프트 빈도 보정("반복 관측=자주, 1회=가끔")으로
  대응 — 표본 2~3개론 빈도 추정이 불가능하기 때문. 표본이 쌓이면(데일리 10-x·미리보기
  수정문) voice_features에 변형 토큰 빈도 통계(사전 대조 또는 Kiwi)를 추가하고
  스타일 게이트가 과다 사용을 감쇠한다. 자유입력("평소에 이렇게 써요")은 사용자의
  습관 선언이므로 항상-급으로 취급.
