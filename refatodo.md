# AMORI — 제품 방향 & 진행

## 제품 방향

AMORI는 사용자의 AI 에이전트가 *먼저* 소개팅을 다녀오는 데이팅 앱이다.
사람이 직접 만나기 전에 두 사람의 에이전트가 대신 대화를 나눠보고, 그 *미리보기*를 제공한다.
태그라인: "AI 에이전트가 먼저 만나는, 가장 나다운 인연."

흐름은 두 단계로 나뉜다.
- **사용자**: 질문지로 자기 페르소나를 만드는 데까지만 한다(가입 → 페르소나).
- **서버**: 그 다음부터는 알아서 한다. 하루 중 여러 번 매칭 → 시뮬레이션 → 리포트를 돌리고,
  사용자는 연결 화면에서 결과를 확인한다.

## 궁합·점수에 대한 원칙 (핵심)

궁합은 결국 **취향**의 문제다. 성향이 반대인 사람에게 끌릴 수도 있고, 상호보완적인 차이가
오히려 더 큰 스파크를 만들기도 한다. 그래서 "비슷하면 좋고 다르면 나쁘다"는 단순한 규칙은 틀렸다.

- 점수는 *유사성*이 아니라 **대화의 케미·스파크**(에너지, 서로 더 알고 싶은 끌림, 다시 만날 이유)를
  본다. 성향이 다르다는 이유만으로 점수를 깎지 않는다.
- 시뮬레이션은 두 사람의 실제 대화 역학을 **정직하게** 재현한다 — 잘 통할 수도, 미지근할 수도,
  한쪽이 계속 밀고 다른 쪽이 닫혀 흥미가 식을 수도 있다. *차이 자체*가 아니라 *역학이 살아 있는가*가 기준.
- 결과물은 판결이 아니라 **미리보기**다. 어디서 통했고 어디서 어긋났는지 신호를 보여주고,
  최종 판단은 사용자의 취향에 맡긴다.
- 케미 게이트(닿지 않은 인연)는 "너무 다른" 매치가 아니라 *스파크가 약한* 미리보기를 거른다.
- 문헌 근거: 사전 설문·명시적 선호만으로 "누가 누구와 통할지"는 예측 불가
  (Eastwick&Finkel 2008, Joel et al. 2017 — ML로도 쌍(dyad) 끌림은 예측 실패).
  그래서 임베딩은 후보 검색까지만 하고 궁합 판정은 시뮬레이션이 한다.
- 같은 이유로 **정답지(받고 싶은 반응)는 매칭 하드필터로 쓰지 않는다** — 리포트의
  소프트 신호("당신이 원하는 방식으로 반응했어요")로만.

## 아키텍처

- **앱(Flutter)**: 화면 + 상태. BFF만 호출한다(LLM 직접 호출·API 키 없음).
- **BFF(FastAPI)**: 모든 도메인 로직 + LLM 호출. **팀 공용 관리형 Postgres** + pgvector가
  단일 원천 (Docker는 2026-07-04부로 제거 — 셋업은 backend/README.md).
- **LLM(provider 스위치)**: `mock`(오프라인 데모) | `gemini` | `modoo`(DevDive 채팅+Gemini 임베딩).
  페르소나 생성·소개팅 시뮬레이션·케미 리포트·임베딩.
- **Firebase**: 인증 + FCM 토큰까지만.

## 핵심 설계

- **소개팅 시뮬레이션 = 원샷 생성.** 양쪽 페르소나를 한 번에 주고 대화 전체를 *한 번의*
  LLM 호출로 만든다(턴마다 호출하지 않음). 서로를 모르는 상태에서 시작한다고 지시해 질문하며
  알아가는 과정을 재현하고, 두 사람이 서로를 이름으로 부르며 각자 다른 말투를 유지한다.
  호출 수·지연이 작아 자동 소개팅을 실서비스에서 돌릴 수 있다.
- **시뮬은 약속을 잡지 않는다 (2026-07-04 결정).** 에이전트 대화는 '알아가기'까지만 —
  잘 맞으면 만나고 싶다는 호감 표현으로 마무리한다. 만남은 두 사용자가 리포트를 보고
  각자 수락하면(수락 게이트 = 리포트 점수 통과) 직접 채팅이 열리고, 약속(날짜·시간)은
  거기서 직접 잡는다.
- **눈치 신호.** 각 발화에 상대 반응 읽기(partner_read) + 전략(알아가기·마무리)이 함께 나온다.
  사용자에겐 발화만 보이고, 신호는 리포트의 내부 근거다.
- **말투는 측정한다.** 주관식 답변에서 코드로 말투 통계(voice_stats)를 뽑아 시뮬 프롬프트에
  수치로 지시하고, 생성된 발화는 **스타일 게이트**가 후처리한다 — 실측에 없는 습관
  (이모지·부호·웃음)의 누출을 결정적으로 제거 (LLM 추측 금지 원칙,
  docs/persona_fidelity_design.md).
- **정답지(받고 싶은 반응) = 평가 전용.** 사용자가 미리 적어둔 "이 상황에서 받고 싶은 답장"은
  리포트 채점(반응성 매칭)에만 쓴다. **시뮬 프롬프트 주입 금지**(상대 에이전트가 정답지에
  맞춰버려 정직한 시뮬이 무너짐), 매칭 하드필터 금지(위 궁합 원칙).
- **자동 소개팅.** 질문지 직후 즉시 시뮬하지 않는다. 서버가 하루 중 여러 번 자동으로 돌린다.
  매칭은 관심 성별이 상호로 맞는 쌍만.
- **시차 송출(라이브 관전).** 한 번에 생성된 대화를 즉시 다 보여주지 않고, 메시지 길이에 비례한
  간격으로 천천히 공개한다(컨베이어벨트처럼). 조회 시점에 공개된 턴만 보이고, 송출이 끝나기
  전에는 점수·게이트 결과를 가린다(스포일러 방지). 공개 시각이 저장돼 재시작에 안전하다.
- **케미 게이트 & 닿지 않은 인연.** 케미 점수가 기준 미만이면 별도 공간으로 분리되고 일정 시간 뒤
  자연 소멸한다. 점수가 낮으면 수락할 수 없고, 진행 중이던 수락도 리셋된다.
- **만남 후 직접 채팅.** 양쪽이 수락해 만남이 확정되면 두 사용자가 직접 대화하며 약속을 잡는다.

## 화면 흐름 (앱)

- **홈**: 에이전트 활동 현황 카드(실데이터). 누르면 연결(진행 중)로 이동.
- **연결**: 진행 중 / 만남 예정 카드. 카드를 누르면 에이전트 대화 + (확정 시) 직접 채팅.
  닿지 않은 인연은 별도 진입.
- **페르소나**: 질문지(객관식 + 말투 주관식) → 페르소나 생성까지.
- **프로필**: 소개팅 가능 일정 입력(직접 약속 조율 참고용 — 시뮬은 일정을 쓰지 않는다).

## 남은 작업

- **Flutter 약속 UI 정리.** 시뮬 약속 폐지(07-04)에 따라 관전 화면의 약속 배지·일정 라벨,
  스케줄링 화면, 일정 시트 잠금 표시를 새 흐름(수락 → 직접 채팅에서 약속)에 맞게 정리.
  백엔드 응답의 appointment_ready는 "수락 가능(리포트 게이트 통과)"으로 의미가 바뀌었고
  appointment_slot은 항상 null이다.
- **페르소나 미리보기·수정 화면 (P0-5).** "당신의 에이전트는 이렇게 말해요" — 수정문이
  최고 등급 말투 데이터로 쌓인다 (persona_fidelity_design.md §6).
- **온보딩·데일리 문항 형식 팀 결정.** 주관식(9-x) 초기 편입 여부, 정답지 수집 문항
  (백엔드는 answer_letter="정답지"로 이미 수신 가능), 미연시/채팅형 연출.
- **리포트 반응성 렌즈 실검증.** 이해·인정·관심 기준과 정답지 대조가 실 LLM에서 의도대로
  작동하는지 E2E + 한국어 대화 품질 평가셋. LLM 저지 블라인드 테스트(말투 충실도)도 병행.
- **FCM 푸시** "에이전트가 다녀왔어요" — 자동 소개팅 완료 시점에 결선. (User에 토큰 컬럼은 있음)
- **더미 매치 졸업.** 실유저 페르소나가 쌓이면 임베딩 매칭으로 전환.
- **매칭 점수 세분화** (가치관/유머/대화 패턴).
- **LLM 유료 티어 전환** — 실유저 PII 유입 전, 무료 티어 학습 사용 방지.
- **레거시 컬럼 정리(후속).** matches.appointment_slot·users.available_slots 기반 예약
  로직(booking)은 시뮬 약속 폐지로 사실상 휴면 — Flutter 정리와 함께 제거 검토.

## 운영 메모

- LLM provider: `.env`의 `LLM_PROVIDER=mock`(쿼터 0, 데모/시드) ↔ `gemini`/`modoo`(실 대화).
- DB는 팀 공용 관리형 Postgres — URL은 시크릿 채널, 마이그레이션은 스키마 변경자가 1회
  (`backend/README.md` 운영 규칙). Docker는 제거됨.
- 시차 송출 on/off·간격, 일일 시뮬 한도는 설정으로 조정한다.
- 로컬 개발 셋업(USB 실기기 `adb reverse` 포함)은 `README.md` 참고.
