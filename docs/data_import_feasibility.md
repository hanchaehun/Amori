# 외부 데이터 import 검토 — 카카오톡 / Spotify / Instagram (2026-06-11)

> AI 에이전트 페르소나(말투·취향) 강화를 위한 import 기능의 기술·법적 타당성 검토.
> 결론: **3개 모두 "사용자가 직접 내려받은 파일을 업로드"하는 단일 아키텍처로 수렴.**
> API 연동 경로는 셋 다 죽었거나(인스타), 원래 없거나(카톡), 정책상 금지(스포티파이).

## ✅ 제품 결정 (2026-06-11, 한채훈)

**카카오톡 import만 진행.** Spotify(점유율 14%+정책 금지)·Instagram(API 폐지)은 보류.
참고할 설계 유의점: 일상 대화 말투와 소개팅 대화 말투는 다르다 — 카톡에서 추출하는 건
"표면 습관"(ㅋㅋ/이모지/문장 길이/부호/말버릇)이고, 소개팅 맥락의 격식 수준(존댓말 등)은
질문지 주관식 9-x(소개팅 상황 메시지 작성)가 잡는다. 두 소스의 역할 분담을 유지할 것 —
카톡 발화를 sample_messages로 그대로 쓰기보다 습관 추출 소스로 쓰는 쪽이 안전.

## 1. 카카오톡 (우선순위 1 — voice의 결정적 소스)

- **내보내기**: 채팅방 → 대화 내용 → 내보내기 → txt를 앱으로 공유. OS별 줄 포맷이 다름(Android/iOS/PC) — 정규식 파싱 난이도 하~중, 오픈소스 파서 존재(graup/kakaotalk-analyzer 등). 사진/이모티콘은 placeholder 텍스트로만 들어옴.
- **법적 (핵심 쟁점)**: 파일에 **상대방 메시지 포함** → 사용자 본인 동의만으로 상대방 개인정보 처리 근거가 안 됨. **이루다 사건(과징금 1억+ 민사 위자료, 민감정보는 3배)이 정확히 이 구조.** 2025 개보위 생성형 AI 안내서도 동일 취지.
- **안전 설계 (필수)**:
  1. **온디바이스 파싱** — 상대방 발화는 단말기에서 폐기, 본인 발화만 서버 전송 (서버가 취득 안 하면 "수집" 불성립 구성)
  2. 본인 발화 안의 제3자 이름 등 개체명 마스킹
  3. 동의 화면에 "상대방 메시지는 기기에서 제거되며 서버로 전송되지 않습니다" 명시
  4. 출시 전 **개보위 사전적정성 검토** 활용 권장 (동종 결정례 존재: "이용자 대화데이터의 AI 모델 학습" 제2025-011-031호)
- 카카오 약관은 내보내기 파일의 타 서비스 업로드를 금지하지 않음. 단 카톡 자동 접근(스크래핑)은 금지 — 반드시 사용자 수동 내보내기 경로만.
- 마이데이터 전송요구권은 메신저 미적용(의료·통신·에너지만) — 적법 근거로 못 씀.

## 2. Spotify (취향 신호 — 말투와 무관)

- **API 경로는 사실상 봉쇄**: 2025-05부터 확장 쿼터는 법인+25만 MAU 요건(닭-달걀), 2026-02부터 개발 모드는 인증 유저 5명 한도.
- **정책 금지 (중요)**: Developer Policy III.14 — *"...or otherwise ingest Spotify Content into a machine learning or AI model"* → **학습뿐 아니라 LLM 프롬프트 투입(inference)도 문언상 금지.** III.13의 유저 프로파일링 금지는 페르소나 생성에 정면 저촉. API로 받은 데이터를 Gemini에 넣는 설계는 불가.
- **대안**: 사용자가 개인정보 설정에서 직접 받은 청취 이력 JSON(Extended Streaming History, 1~14일 소요) 업로드 — Developer Policy(API 이용 규율) 적용 밖 구성 가능.
- **시장 현실**: 한국 음악앱 점유율 유튜브뮤직 42% > 멜론 26% > **스포티파이 14%**. 스포티파이만 지원하면 대다수를 놓침 → "좋아하는 아티스트 직접 입력" 폴백 필요 (멜론/유튜브뮤직은 API 없음).

## 3. Instagram (관심사 + 캡션 말투 약간)

- **개인 계정 API는 2024-12-04 완전 폐지** (Basic Display API). 후속 API는 비즈니스/크리에이터 계정 전용 — 일반 사용자 대상 데이팅 앱 UX로 비현실적.
- **유일 경로 = DYI 파일**: Accounts Center → 정보 다운로드 → JSON 선택(날짜 범위·데이터 유형 선택 가능, ~48시간 소요). 게시물 캡션·댓글·DM·스토리 포함. 주의: JSON 한글이 이중 인코딩(latin-1 디코딩 필요).
- **법적**: Meta 약관은 사용자의 DYI 제3자 업로드를 금지하지 않음 (DYI 자체가 데이터 이동권 대응 기능). 단 댓글 작성자·DM 상대방 등 **제3자 데이터는 카톡과 동일 구조** → 본인 캡션/게시물만 클라이언트에서 추출.

## 공통 — 데이팅 앱 가중 리스크

- **민감정보(PIPA 23조)**: 매칭 선호 성별 = 성적 지향 = 민감정보. 대화/취향에서 *추론*되는 것 포함 **분리된 별도 동의** 필수. 이루다 민사에서 민감정보 위자료가 일반의 3배.
- **동의 화면 요건**: 수집 항목 구체 특정("본인이 작성한 메시지"), 목적 구체 특정("말투 반영 AI 에이전트 생성"), AI 활용 여부 명시, 보유 기간·파기, 거부권, 국외 LLM API(Gemini) 처리위탁·국외이전 고지. "신규 서비스 개발" 같은 포괄 기재는 이루다에서 무효 판정.

## 구현 시사점

1. 셋 다 **"파일 업로드 → 클라이언트 파싱 → 본인 데이터만 서버"** 단일 파이프라인 — 컴포넌트 재사용 가능
2. 우선순위: **카톡(말투 금광) > Spotify(매칭 취향) ≈ Instagram(관심사)**. voice 강화 목적이면 카톡만으로 충분
3. 카톡 import가 모이면 per-user LoRA + 셀프호스팅(한국어 모델) 장기 옵션과 연결됨 (refatodo 참고)
4. 출시 전 법무 검토 + 개보위 사전적정성 검토 — 본 문서는 법률 자문이 아님
