# 페르소나 충실도(voice fidelity) 설계 — "가장 나다운 에이전트" (2026-07-04)

> 목표: 에이전트가 사용자의 **실제 성격(매칭 신호)과 말투(voice)**를 정교하게 재현한다.
> 결론 요약: 객관식은 성향(매칭)을 잡지만 말투는 못 잡는다(실 Gemini E2E에서 실증, scenarios.dart 주석).
> 현재 초기 온보딩은 객관식 5문항뿐이라 **voice가 100% LLM 추측으로 시작**한다 — 이것이 1순위 문제.
> 해법은 "질문 수 늘리기"가 아니라 **① 자유 발화 수집을 앞당기고 ② 말투를 측정값으로 만들고
> ③ 프롬프트 재현을 측정값 기반으로 바꾸고 ④ 충실도를 검증하는** 4레이어 파이프라인.

## 0. 전제 — 임베딩과 voice는 다른 문제

| | 매칭 임베딩 | voice(말투 모사) |
|---|---|---|
| 무엇 | traits·가치관 텍스트 → 1024차원 벡터 (`persona_embedding_text`) | `speech_style` + `sample_messages` → 시뮬 프롬프트 컨디셔닝 (`_speech_block`) |
| 소비처 | pgvector 후보 검색 (누구와 시뮬할지) | 시뮬레이션 발화 생성 (어떻게 말할지) |
| 현재 한계 | 단일 벡터, speech_style 미포함(의도됨 — 말투 유사가 궁합 아님) | few-shot 최대 3개, 스타일이 3지선다 enum 수준, 측정 아닌 LLM 추측 |

"임베딩을 정교하게"의 실체는 두 갈래다: **매칭 벡터의 의미 표면 강화**(7절)와 **voice 파이프라인 재설계**(3~6절).
사용자가 체감하는 "나 같다"는 거의 전부 후자에서 나온다.

## 1. 현재 파이프라인 진단

- **초기**: R-1~R-5 객관식 5문항 → `build_persona`. 주관식 0개 → sample_messages를 LLM이 창작
  (2026-07-04 modoo 힌트 수정으로 빈 배열 실패는 막았지만, "지어낸 목소리"라는 본질은 동일).
- **데일리**: 하루 1문항, `kDailyScenarioCodes` 순서상 9-1~9-3(주관식)이 먼저 오지만
  **실제 말투 데이터가 3일 뒤에야 3개 확보**된다. 그동안 에이전트는 추측 말투로 소개팅을 다닌다.
- **재현**: `_speech_block`이 enum 스타일 카드 1줄 + 샘플 ≤3개. 문장 길이 "보통", 이모지 "가끔" 같은
  coarse 값으로는 "ㅋㅋ를 몇 개 붙이는지, ~를 어디 쓰는지, 띄어쓰기를 뭉개는지"가 재현 안 됨.
- **검증**: 생성된 발화가 사용자 말투와 닮았는지 측정하는 장치가 전무. 사용자가 고칠 방법도 없음.

## 2. 설계 원칙

1. **성향은 객관식, 말투는 자유 발화** — 역할 분담 유지 (data_import_feasibility.md 제품 결정과 일치).
2. **말투는 추측하지 말고 측정한다** — 자유 발화에서 코드로 통계를 뽑는다(LLM 추론은 보조).
3. **출처(provenance)를 기록한다** — 사용자가 직접 쓴 문장 > 카톡 추출 > LLM 생성. 실데이터가
   쌓이면 낮은 등급을 자동 대체.
4. **충실도를 게이트로 건다** — voice 신뢰도가 낮으면(실측 샘플 < N) 미리보기·수정으로 보강 후 매칭 투입.

## 3. Layer 1 — 수집: 문항 개편 + 자유 발화 뱅크

### 3-1. 온보딩 개편 (P0, 가장 급함)

`kInitialScenarioCodes = [R-1..R-5]` → `[R-1..R-5, 9-1, 9-2]` (+선택 9-3).
9-x는 이미 존재하고 `isFreeText` 렌더링도 플레이어에 구현돼 있으므로 **코드 변경은 리스트 한 줄**.
9-x 3개는 서로 다른 말투 단면을 트리거하도록 설계돼 있다(난처함 대처 / 취향 티키타카 / 칭찬 반응).
온보딩 이탈이 걱정되면 9-1·9-2만 필수, 9-3은 "말투 정확도 올리기" 선택 카드로.

효과: **가입 직후부터 진짜 발화 2~3개 확보** → sample_messages가 창작이 아닌 실문장으로 시작.

### 3-2. 데일리 문항 재편성 (P0)

하루 1개 페이스는 유지하되(질문 피로 방지), 큐를 "객관식:주관식 = 2:1 인터리브"로 재배열하고
주관식 뱅크를 신설한다. 신규 주관식(10-x) — **레지스터(상황별 말투 단면) 커버리지**가 설계 기준:

| 코드 | 레지스터 | 상황 (사용자가 실제 답장을 작성) |
|---|---|---|
| 10-1 | 오프닝/아침 톤 | 어제 처음 연락한 상대에게 아침에 먼저 보내는 첫 메시지 |
| 10-2 | 클로징/밤 톤 | 늦은 밤 대화를 기분 좋게 마무리하는 마지막 메시지 |
| 10-3 | 약속 제안 | 마음에 드는 상대에게 주말에 보자고 먼저 제안 |
| 10-4 | 완곡 거절 | 이번 주는 어려운데 관계는 이어가고 싶을 때의 거절 답장 |
| 10-5 | 지각 사과 | 약속에 15분 늦을 것 같을 때 보내는 메시지 |
| 10-6 | 축하 리액션 | 상대가 "나 붙었어!!"라고 합격 소식을 보냈을 때 |
| 10-7 | 위로 | 상대가 "오늘 일 진짜 힘들었다…"라고 보냈을 때 |
| 10-8 | 서운함 표현 | 상대 답장이 이틀째 뜸할 때 보내는(또는 안 보내는) 메시지 |
| 10-9 | 농담 응수 | 상대가 가벼운 드립을 쳤을 때 받아치는 답 |
| 10-10 | 취향 논쟁 | "민초는 좀 아니지 않아요?"에 대한 반응(동의든 반박이든) |
| 10-11 | 자기 근황 | "주말에 뭐 했어요?"에 사진 없이 말로만 답하기 |
| 10-12 | 부탁 | 상대에게 작은 부탁(맛집 추천 등)을 하는 메시지 |

포인트: 시뮬레이션 대화가 실제로 지나가는 국면(오프닝→알아가기→리액션→약속 조율→마무리)과
1:1로 대응시켰다. 재현 단계(5절)에서 **국면에 맞는 샘플을 골라 넣기 위한 태그**가 된다.

### 3-3. 카카오톡 import (P1 — 이미 제품 결정됨)

data_import_feasibility.md 결정 그대로: 온디바이스 파싱 → 본인 발화만 서버 → **습관 추출 소스**
(sample_messages 원문 직행은 지양 — 일상 말투 ≠ 소개팅 말투). 4절의 측정 통계 입력으로 쓰면
표본이 수백~수천 개로 뛰어 통계 신뢰도가 완성된다. 격식(존댓말 여부)은 9-x/10-x가 잡는다.

### 3-4. 운영 중 수집 (P2)

- **시뮬 턴 피드백**: 관전 화면에서 내 에이전트 발화에 "나 같지 않아요" 롱프레스 → 해당 발화가
  negative anchor로 저장, 다음 시뮬 프롬프트에 "이렇게 말하지 마세요" 예시로 주입.
- **만남 후 직접 채팅**: 본인 발화는 1st-party 데이터 — 별도 동의 후 습관 통계 갱신에 사용
  (민감정보·동의 요건은 feasibility 문서 공통 절 준수).

## 4. Layer 2 — 표현: voice profile v2

`persona.schema.json` 확장 (기존 필드 유지 + 추가, 하위호환):

```jsonc
{
  "speech_style": { /* 기존 8필드 유지 — LLM 추론 요약 */ },
  "voice_stats": {            // ★ 신규 — 코드가 계산한 측정값 (LLM 추측 금지)
    "sample_count": 14,        // 실측 발화 수 = 신뢰도의 근거
    "formality_ratio": {"존댓말": 0.9, "반말": 0.1},   // 어미 분석(-요/-니다/-어)
    "len_chars": {"p25": 12, "p50": 21, "p75": 38},
    "laugh": {"token": "ㅋ", "avg_run": 3.2, "per_msg": 0.6},
    "emoji": {"per_msg": 0.0, "inventory": []},
    "punct_per_msg": {"~": 0.4, "!!": 0.1, "…": 0.2, "ㅠㅠ": 0.3},
    "question_ratio": 0.35,
    "interjections": ["헉", "아 맞다"],
    "spacing": "정확"          // 띄어쓰기: 정확|보통|뭉갬
  },
  "sample_bank": [             // ★ 신규 — sample_messages(1~3) 대체·확장
    {"text": "...", "register": "위로", "source": "user_written",  // user_written|kakao|llm_seed
     "collected_at": "2026-07-05"}
  ],
  "voice_confidence": 0.42     // sample_count·레지스터 커버리지에서 계산
}
```

- 신규 모듈 `backend/app/llm/voice_features.py`: `extract_voice_stats(texts) -> dict`. 순수 함수 +
  정규식/카운팅이라 LLM 무관, 단위테스트 용이. build/update_persona가 주관식 답변을 여기 통과시킴.
- `sample_messages`(1~3)는 시뮬 하위호환용으로 유지하되 sample_bank에서 파생(대표 3개).
- `source=llm_seed`(창작 샘플)는 실측이 들어오는 즉시 교체 — 현 modoo 힌트의 창작은 "부트스트랩
  임시값"으로 지위가 명확해진다.

## 5. Layer 3 — 재현: 프롬프트 컨디셔닝 v2

`_speech_block` 개편 (simulation.py):

1. **측정값 스타일 카드** — enum 대신 수치 지시:
   `말투(측정값 — 반드시 재현): 존댓말 90% | 문장 12~38자(중앙 21자) | ㅋ 3~4연속·메시지 절반에 |
   이모지 절대 안 씀 | ~ 자주, ㅠㅠ 가끔 | 3번에 1번은 되묻기 | 감탄사 "헉"`
2. **레지스터 매칭 few-shot** — 항상 같은 3개 대신, sample_bank에서 대화 국면별로 5~8개 선별
   (오프닝 1 + 리액션 2 + 티키타카 2 + 약속/마무리 1…). 원샷 시뮬은 전 국면을 한 콜에 생성하므로
   레지스터별 1개씩 골고루 넣는 것이 효율적.
3. **금지 규칙 명시** — inventory가 빈 항목은 포지티브 지시보다 강한 네거티브로:
   "이모지·~를 절대 쓰지 마세요"(사용자가 안 쓰는 습관의 누출이 '나 같지 않음'의 최대 원인).
4. **스타일 게이트 (후검증)** — 시뮬 생성 후 각 발화를 `voice_features`로 재측정, 페르소나 통계와
   대조: 격식 위반 / 미보유 이모지 사용 / 길이 z-score>2 다수 → 1회 재생성, 경미하면 후편집
   (이모지 제거 등). 기존 "약속 슬롯 교집합 재검증"과 같은 자리(엔진 후처리)에 들어간다.

## 6. Layer 4 — 평가·피드백 루프

- **페르소나 미리보기 (P0, 체감 효과 최대)**: 페르소나 생성 직후 "당신의 에이전트는 이렇게
  말해요" — 상황 3개(인사/리액션/거절)에 대한 에이전트 발화를 보여주고 사용자가 직접 수정 가능.
  수정문은 `source=user_written`으로 sample_bank에 추가(= 최고 등급 데이터가 공짜로 쌓임).
- **자동 충실도 지표**: ① 결정적 — 생성 발화 vs voice_stats 거리(격식 일치율, 길이 분포, 부호
  재현율) ② LLM 저지 블라인드 테스트 — 실발화 1 + 에이전트 발화 1을 주고 어느 쪽이 진짜인지
  맞히게 함. **정답률이 우연(50%)에 가까울수록 모사 성공** — 60% 이하를 목표선으로.
- **voice_confidence 게이트**: 실측 샘플 < 3이면 자동 매칭 투입 전 미리보기 보강을 유도.
  (데이터 사다리: 0개=순수 추측 → 3개=격식·웃음 잡힘 → 10~20개=통계 안정 → 카톡=완성)

## 7. 매칭 임베딩 정교화 (별도 트랙, 소규모)

- `persona_embedding_text` 의미 표면 강화: traits 요약 외에 데일리 객관식의 **원답변 누적**을 축약
  반영(정보 손실 감소). R-1(관계 목적)은 매칭 하드필터 후보(목적 불일치는 시뮬 전에 거르기).
- refatodo 원칙 유지 — 점수는 유사성이 아니라 케미. 임베딩은 후보 검색까지만, 궁합 판정은 시뮬이
  한다. 따라서 다중 벡터(가치관/유머/생활) 분리는 실유저 데이터로 검색 품질 문제가 확인된 뒤(P2).

## 8. 실행 계획

| 단계 | 내용 | 주요 변경 파일 |
|---|---|---|
| **P0-1** | 온보딩에 9-1·9-2 편입 | `scenarios.dart` (kInitialScenarioCodes 한 줄) |
| **P0-2** | `voice_features.py` 측정 모듈 + 단위테스트 | backend 신규 |
| **P0-3** | 스키마 v2 (voice_stats·sample_bank·provenance) | `persona.schema.json`, `output_schemas.py`, `schemas/persona.py`, alembic |
| **P0-4** | `_speech_block` v2 (측정값 카드 + 네거티브 규칙) | `prompts/simulation.py` |
| **P0-5** | 페르소나 미리보기·수정 화면 | Flutter `features/persona/` + BFF 엔드포인트 |
| **P1-1** | 데일리 주관식 뱅크 10-x 12문항 + 큐 인터리브 | `scenarios.dart` |
| **P1-2** | 레지스터 매칭 few-shot 선별 + 스타일 게이트 | `simulation.py`, 시뮬 엔진 후처리 |
| **P1-3** | 카톡 import (온디바이스 파싱, feasibility 문서 준수) | Flutter + BFF 신규 |
| **P2** | 시뮬 턴 피드백, 직접 채팅 마이닝(동의), LLM 저지 대시보드, 다중 벡터 검토 | — |

P0만으로 "가입 직후 실측 말투 2~3개 + 측정 기반 재현 + 사용자 수정 루프"가 성립한다 —
현재의 "3일간 지어낸 목소리로 소개팅" 문제가 구조적으로 사라진다.
