# 페르소나 충실도(voice fidelity) 설계 — "가장 나다운 에이전트" (2026-07-04)

> 목표: 에이전트가 사용자의 **실제 성격(매칭 신호)과 말투(voice)**를 정교하게 재현한다.
> 결론 요약: 객관식은 성향(매칭)을 잡지만 말투는 못 잡는다(실 Gemini E2E에서 실증, scenarios.dart 주석).
> 현재 초기 온보딩은 객관식 5문항뿐이라 **voice가 100% LLM 추측으로 시작**한다 — 이것이 1순위 문제.
> 해법은 "질문 수 늘리기"가 아니라 **① 자유 발화 수집을 앞당기고 ② 말투를 측정값으로 만들고
> ③ 프롬프트 재현을 측정값 기반으로 바꾸고 ④ 충실도를 검증하는** 4레이어 파이프라인.

## 0. 전제 — 임베딩과 voice는 다른 문제

| | 매칭 임베딩 | voice(말투 모사) |
|---|---|---|
| 무엇 | traits·가치관 텍스트 → 1024차원 벡터 (`persona_embedding_text`) | `speech_style` + `sample_messages` → 시뮬 프롬프트 컨디셔닝 (`_speech_block`) |
| 소비처 | pgvector 후보 검색 (누구와 시뮬할지) | 시뮬레이션 발화 생성 (어떻게 말할지) |
| 현재 한계 | 단일 벡터, speech_style 미포함(의도됨 — 말투 유사가 궁합 아님) | few-shot 최대 3개, 스타일이 3지선다 enum 수준, 측정 아닌 LLM 추측 |

"임베딩을 정교하게"의 실체는 두 갈래다: **매칭 벡터의 의미 표면 강화**(7절)와 **voice 파이프라인 재설계**(3~6절).
사용자가 체감하는 "나 같다"는 거의 전부 후자에서 나온다.

## 1. 현재 파이프라인 진단

- **초기**: R-1~R-5 객관식 5문항 → `build_persona`. 주관식 0개 → sample_messages를 LLM이 창작
  (2026-07-04 modoo 힌트 수정으로 빈 배열 실패는 막았지만, "지어낸 목소리"라는 본질은 동일).
- **데일리**: 하루 1문항, `kDailyScenarioCodes` 순서상 9-1~9-3(주관식)이 먼저 오지만
  **실제 말투 데이터가 3일 뒤에야 3개 확보**된다. 그동안 에이전트는 추측 말투로 소개팅을 다닌다.
- **재현**: `_speech_block`이 enum 스타일 카드 1줄 + 샘플 ≤3개. 문장 길이 "보통", 이모지 "가끔" 같은
  coarse 값으로는 "ㅋㅋ를 몇 개 붙이는지, ~를 어디 쓰는지, 띄어쓰기를 뭉개는지"가 재현 안 됨.
- **검증**: 생성된 발화가 사용자 말투와 닮았는지 측정하는 장치가 전무. 사용자가 고칠 방법도 없음.

## 2. 설계 원칙

1. **성향은 객관식, 말투는 자유 발화** — 역할 분담 유지 (2026-06-11 제품 결정과 일치, 3-3절 참조).
2. **말투는 추측하지 말고 측정한다** — 자유 발화에서 코드로 통계를 뽑는다(LLM 추론은 보조).
3. **출처(provenance)를 기록한다** — 사용자가 직접 쓴 문장 > 카톡 추출 > LLM 생성. 실데이터가
   쌓이면 낮은 등급을 자동 대체.
4. **충실도를 게이트로 건다** — voice 신뢰도가 낮으면(실측 샘플 < N) 미리보기·수정으로 보강 후 매칭 투입.

## 3. Layer 1 — 수집: 문항 개편 + 자유 발화 뱅크

### 3-1. 온보딩 개편 (P0 제안 — ⚠️ 미확정, 팀 결정 필요)

> 현재 "대표 질문 5개 + 하루 1문항" 구조는 명세현 작업(09fc86f, 4e570c5)이다.
> 아래는 제안이며, 온보딩 문항 구성 변경 여부는 팀에서 결정한다.

제안: `kInitialScenarioCodes = [R-1..R-5]` → `[R-1..R-5, 9-1, 9-2]` (+선택 9-3).
9-x는 이미 존재하고 `isFreeText` 렌더링도 플레이어에 구현돼 있으므로 **코드 변경은 리스트 한 줄**.
9-x 3개는 서로 다른 말투 단면을 트리거하도록 설계돼 있다(난처함 대처 / 취향 티키타카 / 칭찬 반응).
온보딩 이탈이 걱정되면 9-1·9-2만 필수, 9-3은 "말투 정확도 올리기" 선택 카드로.

효과: **가입 직후부터 진짜 발화 2~3개 확보** → sample_messages가 창작이 아닌 실문장으로 시작.

### 3-2. 데일리 문항 재편성 (P0)

하루 1개 페이스는 유지하되(질문 피로 방지), 큐를 "객관식:주관식 = 2:1 인터리브"로 재배열하고
주관식 뱅크를 신설한다. 신규 주관식(10-x) — **레지스터(상황별 말투 단면) 커버리지**가 설계 기준:

| 코드 | 레지스터 | 상황 (사용자가 실제 답장을 작성) |
|---|---|---|
| 10-1 | 오프닝/아침 톤 | 어제 처음 연락한 상대에게 아침에 먼저 보내는 첫 메시지 |
| 10-2 | 클로징/밤 톤 | 늦은 밤 대화를 기분 좋게 마무리하는 마지막 메시지 |
| 10-3 | 약속 제안 | 마음에 드는 상대에게 주말에 보자고 먼저 제안 |
| 10-4 | 완곡 거절 | 이번 주는 어려운데 관계는 이어가고 싶을 때의 거절 답장 |
| 10-5 | 지각 사과 | 약속에 15분 늦을 것 같을 때 보내는 메시지 |
| 10-6 | 축하 리액션 | 상대가 "나 붙었어!!"라고 합격 소식을 보냈을 때 |
| 10-7 | 위로 | 상대가 "오늘 일 진짜 힘들었다…"라고 보냈을 때 |
| 10-8 | 서운함 표현 | 상대 답장이 이틀째 뜸할 때 보내는(또는 안 보내는) 메시지 |
| 10-9 | 농담 응수 | 상대가 가벼운 드립을 쳤을 때 받아치는 답 |
| 10-10 | 취향 논쟁 | "민초는 좀 아니지 않아요?"에 대한 반응(동의든 반박이든) |
| 10-11 | 자기 근황 | "주말에 뭐 했어요?"에 사진 없이 말로만 답하기 |
| 10-12 | 부탁 | 상대에게 작은 부탁(맛집 추천 등)을 하는 메시지 |

포인트: 대화가 실제로 지나가는 국면(오프닝→알아가기→리액션→호감 표현→마무리)과 1:1로
대응시켰다. 재현 단계(5절)에서 **국면에 맞는 샘플을 골라 넣기 위한 태그**가 된다.
(10-3~10-5 같은 약속·일정 레지스터는 시뮬이 아니라 **수락 후 직접 채팅**에서 쓰는 말투다 —
시뮬은 약속을 잡지 않는다. 07-04 결정.)

### 3-3. 카카오톡 import (P1 — 2026-06-11 제품 결정, 한채훈)

> 상세 타당성 검토(카톡/Spotify/Instagram 비교, 법적 분석)는 git 이력의
> `docs/data_import_feasibility.md`(009bd22) 참조. 핵심 결정만 여기 남긴다.

- **카톡만 진행.** Spotify(점유율 14% + API 데이터의 LLM 투입은 정책 금지)·Instagram(개인 API
  2024-12 폐지)은 보류. 셋 다 유일 경로는 "사용자가 직접 내려받은 파일 업로드"로 동일했다.
- **아키텍처(법적 필수)**: 사용자가 수동 내보낸 txt 업로드 → **온디바이스 파싱, 본인 발화만
  서버 전송**(상대방 메시지는 단말에서 폐기 — 상대방 동의 없는 수집 = 이루다 사건 구조).
  본인 발화 속 제3자 이름 등 개체명 마스킹. 카톡 자동 접근(스크래핑)은 약관 금지 — 수동 내보내기만.
- **용도**: sample_messages 원문 직행이 아니라 **습관 추출 소스**(일상 말투 ≠ 소개팅 말투;
  격식 수준은 9-x/10-x가 잡음). voice_stats 표본이 수백~수천 개로 뛰어 통계가 완성된다.
- **동의 화면 요건**: "상대방 메시지는 기기에서 제거되며 전송되지 않습니다" 명시, 수집
  항목·목적("말투 반영 에이전트 생성")·국외 LLM 처리위탁 구체 고지, 매칭 선호 성별=민감정보
  **분리 동의**. 출시 전 개보위 사전적정성 검토 권장. 포괄 기재("신규 서비스 개발")는 무효 선례.

### 3-4. 운영 중 수집 (P2)

- **시뮬 턴 피드백**: 관전 화면에서 내 에이전트 발화에 "나 같지 않아요" 롱프레스 → 해당 발화가
  negative anchor로 저장, 다음 시뮬 프롬프트에 "이렇게 말하지 마세요" 예시로 주입.
- **만남 후 직접 채팅**: 본인 발화는 1st-party 데이터 — 별도 동의 후 습관 통계 갱신에 사용
  (민감정보·동의 요건은 feasibility 문서 공통 절 준수).

## 4. Layer 2 — 표현: voice profile v2

`persona.schema.json` 확장 (기존 필드 유지 + 추가, 하위호환):

```jsonc
{
  "speech_style": { /* 기존 8필드 유지 — LLM 추론 요약 */ },
  "voice_stats": {            // ★ 신규 — 코드가 계산한 측정값 (LLM 추측 금지)
    "sample_count": 14,        // 실측 발화 수 = 신뢰도의 근거
    "formality_ratio": {"존댓말": 0.9, "반말": 0.1},   // 어미 분석(-요/-니다/-어)
    "len_chars": {"p25": 12, "p50": 21, "p75": 38},
    "laugh": {"token": "ㅋ", "avg_run": 3.2, "per_msg": 0.6},
    "emoji": {"per_msg": 0.0, "inventory": []},
    "punct_per_msg": {"~": 0.4, "!!": 0.1, "…": 0.2, "ㅠㅠ": 0.3},
    "question_ratio": 0.35,
    "interjections": ["헉", "아 맞다"],
    "spacing": "정확"          // 띄어쓰기: 정확|보통|뭉갬
  },
  "sample_bank": [             // ★ 신규 — sample_messages(1~3) 대체·확장
    {"text": "...", "register": "위로", "source": "user_written",  // user_written|kakao|llm_seed
     "collected_at": "2026-07-05"}
  ],
  "voice_confidence": 0.42     // sample_count·레지스터 커버리지에서 계산
}
```

- 신규 모듈 `backend/app/llm/voice_features.py`: `extract_voice_stats(texts) -> dict`. 순수 함수 +
  정규식/카운팅이라 LLM 무관, 단위테스트 용이. build/update_persona가 주관식 답변을 여기 통과시킴.
- `sample_messages`(1~3)는 시뮬 하위호환용으로 유지하되 sample_bank에서 파생(대표 3개).
- `source=llm_seed`(창작 샘플)는 실측이 들어오는 즉시 교체 — 현 modoo 힌트의 창작은 "부트스트랩
  임시값"으로 지위가 명확해진다.

## 5. Layer 3 — 재현: 프롬프트 컨디셔닝 v2

`_speech_block` 개편 (simulation.py):

1. **측정값 스타일 카드** — enum 대신 수치 지시:
   `말투(측정값 — 반드시 재현): 존댓말 90% | 문장 12~38자(중앙 21자) | ㅋ 3~4연속·메시지 절반에 |
   이모지 절대 안 씀 | ~ 자주, ㅠㅠ 가끔 | 3번에 1번은 되묻기 | 감탄사 "헉"`
2. **레지스터 매칭 few-shot** — 항상 같은 3개 대신, sample_bank에서 대화 국면별로 5~8개 선별
   (오프닝 1 + 리액션 2 + 티키타카 2 + 약속/마무리 1…). 원샷 시뮬은 전 국면을 한 콜에 생성하므로
   레지스터별 1개씩 골고루 넣는 것이 효율적.
3. **금지 규칙 명시** — inventory가 빈 항목은 포지티브 지시보다 강한 네거티브로:
   "이모지·~를 절대 쓰지 마세요"(사용자가 안 쓰는 습관의 누출이 '나 같지 않음'의 최대 원인).
4. **스타일 게이트 (후검증)** — ✅ v1 구현됨(2026-07-04, `services/style_gate.py`):
   미보유 이모지·부호(~/ㅠㅠ/!!/^^/;;/…)·웃음 습관을 결정적으로 제거/교정하는 후편집,
   격식 위반은 로깅(재생성 트리거의 근거 데이터로 축적). 재생성(1회)은 쿼터 비용 때문에
   보류 — 위반 로그가 쌓이면 도입 판단. 두 시뮬 경로(수동 SSE·자동) 모두 통과한다.

## 6. Layer 4 — 평가·피드백 루프

- **페르소나 미리보기 (P0, 체감 효과 최대)**: 페르소나 생성 직후 "당신의 에이전트는 이렇게
  말해요" — 상황 3개(인사/리액션/거절)에 대한 에이전트 발화를 보여주고 사용자가 직접 수정 가능.
  수정문은 `source=user_written`으로 sample_bank에 추가(= 최고 등급 데이터가 공짜로 쌓임).
- **자동 충실도 지표**: ① 결정적 — 생성 발화 vs voice_stats 거리(격식 일치율, 길이 분포, 부호
  재현율) ② LLM 저지 블라인드 테스트 — 실발화 1 + 에이전트 발화 1을 주고 어느 쪽이 진짜인지
  맞히게 함. **정답률이 우연(50%)에 가까울수록 모사 성공** — 60% 이하를 목표선으로.
- **voice_confidence 게이트**: 실측 샘플 < 3이면 자동 매칭 투입 전 미리보기 보강을 유도.
  (데이터 사다리: 0개=순수 추측 → 3개=격식·웃음 잡힘 → 10~20개=통계 안정 → 카톡=완성)

## 7. 매칭 임베딩 정교화 (별도 트랙, 소규모)

- `persona_embedding_text` 의미 표면 강화: traits 요약 외에 데일리 객관식의 **원답변 누적**을 축약
  반영(정보 손실 감소). R-1(관계 목적)은 매칭 하드필터 후보(목적 불일치는 시뮬 전에 거르기).
- refatodo 원칙 유지 — 점수는 유사성이 아니라 케미. 임베딩은 후보 검색까지만, 궁합 판정은 시뮬이
  한다. 따라서 다중 벡터(가치관/유머/생활) 분리는 실유저 데이터로 검색 품질 문제가 확인된 뒤(P2).
- **벡터 DB 현황과 진화 로드맵** (구 matching/README.md에서 이관): pgvector 1024차원 HNSW 코사인
  인덱스, `backend/app/matching/ranker.py`가 top-K 코사인 단일 쿼리 베이스라인(User 조인 포함).
  로드맵 ① 카테고리별 부분 점수(가치관/유머/대화 패턴 — match_list 필터의 실데이터화)
  ② 시뮬레이션 오케스트레이션(어떤 쌍을 언제, 사용자당 매일 N건) ③ 만남 후 feedback 테이블을
  매칭 점수에 반영하는 학습 루프. 매칭 점수 산식 변경 시 백엔드 리뷰 필수.

## 9. 심리학 + 최신 기법 레이어 (2026-07 리서치)

### 9-1. 말투의 3층 모델 — NLP는 1층만 본다

| 층 | 내용 | 담당 |
|---|---|---|
| 1층 표면 습관 | 어미·존대, ㅋㅋ/이모지/부호, 문장 길이, 말버릇 | voice_features 측정 (4절) |
| 2층 화용 행동 | 질문 비율, 자기개방 속도, 리액션 크기, 화제 전환, 갈등 시 접근/회피 | ★ conversation_policy 신설 |
| 3층 심리 기저 | Big Five, 애착 성향 — 2층 행동을 생성하는 원인 | ★ 심리 프로파일 신설 |

"나 같다"는 1층만으로 안 된다 — 존댓말·ㅋㅋ가 같아도 **서운할 때 반응**이 다르면 남이다.

### 9-2. 심리학에서 가져올 것

- **기능어 스타일로메트리** (Pennebaker/LIWC 계열): 스타일 지문은 내용어가 아니라 기능어에 있다.
  한국어에선 **어미·조사 선택**이 그 역할 — voice_features에 어미 분포(-요/-용/-염/-슴/-지/-잖아)를
  형태소 분석기(Kiwi)로 추가하면 측정 해상도가 크게 오른다.
- **Big Five ↔ 언어 마커** (Mairesse 2007 → PersonaLLM NAACL'24 → PsyPlay'25): 외향성→발화량·긍정
  정서·감탄, 신경성→부정 정서·1인칭 단수, 친화성→완화 표현·맞장구. LLM이 트레이트 지시를 언어
  마커로 변환·표현할 수 있음은 반복 검증됨. → 페르소나에 big_five 5축(점수+판단 근거 답변)을 저장하고
  시뮬 프롬프트에 "트레이트→대화 행동" 매핑을 지시한다.
- **애착 이론 (데이팅 도메인 핵심)**: 불안 애착→답장 지연에 민감·재확인 추구, 회피 애착→자기개방
  낮음·거리 조절. **R-2(연락)·R-3(서운함)·R-4(위로)가 이미 애착 축과 겹친다** — ECR 단축형 문항과
  명시 매핑하면 "서운한 상황에서 에이전트가 어떻게 구는지"가 심리적으로 일관돼진다. 시뮬의
  partner_read/strategy(눈치 신호)에 애착 성향을 조건으로 걸 수 있다.
- **사회침투 이론 (Altman-Taylor)**: 자기개방의 깊이·속도는 개인차가 크고 관계 진전 체감을 만든다.
  R-1(관계 목적)·6번대(관계 속도) 답변 → 자기개방 속도 파라미터로.
- **조율(accommodation)/LSM 주의점**: 실제 사람은 상대에게 어느 정도 수렴한다. 현행 "상대 말투를
  따라가지 마세요"는 style bleed 방지로 옳지만 과하면 부자연 — "화제·energy는 수렴 가능,
  어미·습관·이모지는 고정"으로 수렴 허용 범위를 명시한다.
- **프라이버시 원칙**: 추론된 심리 프로파일은 사용자에게 공개·수정권 부여(미리보기와 결합),
  리포트에서 단정적 성격 판정 문구 금지. 성적 지향 외 심리 추론도 프로파일링 고지 대상으로 취급.

### 9-3. 최신 기법 — 현실성 등급 (DevDive API = 가중치·활성값 접근 불가 전제)

| 기법 | 근거 | AMORI 적용 | 등급 |
|---|---|---|---|
| **인터뷰 기반 에이전트** | Stanford 1,052명: 2시간 인터뷰→2주 뒤 응답 85% 재현 (HAI '25). 설문보다 인터뷰가 우월 | 온보딩을 "내 에이전트와의 첫 채팅"으로 — 객관식은 채팅 속 칩 선택, 주관식은 자연 답장. **답장 자체가 말투 표본 + 내용이 성격 데이터** → 질문 피로 문제까지 동시 해결 | **A (강추, 문항 구조 팀 결정과 연계)** |
| few-shot ICL의 한계 인지 | "Catch Me If You Can" '25: 샘플 2~5개 ICL은 평균 스타일로 회귀 — 일반인 모사는 미해결 | 샘플 늘리기(카톡)+측정값 명시 지시+**생성 후 스타일 후편집**(post-editing '26 연구 방향)이 정확한 보완 — 5절 게이트 강화 | A (P0/P1에 이미 반영) |
| 스타일 임베딩 채점 | 스타일 임베딩 vs LLM 저지 벤치마크 ('25) — 둘의 삼각측량이 표준 | 생성문↔사용자 스타일 벡터 코사인으로 충실도 자동 채점. 단 한국어 스타일 임베딩 부재 → 초기엔 voice_features+LLM 저지로 대체, 데이터 모이면 자체 소규모 학습 | B (P2) |
| 퍼스나 벡터/활성값 스티어링 | Anthropic persona vectors '25 | API 모델에선 불가. 셀프호스팅 한국어 오픈모델 전환 시에만 | C (장기) |
| per-user LoRA | refatodo 장기 옵션과 동일 | 카톡 데이터 축적 + 셀프호스팅 전제. 유저당 비용·서빙 복잡도 커서 스타일 카드+few-shot로 먼저 한계 확인 후 | C (장기) |

### 9-4. 스키마 v2 추가분

```jsonc
{
  "psych_profile": {          // ★ 3층 — 사용자 공개·수정 가능
    "big_five": {"E": 0.3, "A": 0.7, "C": 0.6, "N": 0.4, "O": 0.5, "evidence": ["R-2:A", "..."]},
    "attachment_hint": "안정-약간 불안",   // 단정 금지, hint로만
    "confidence": 0.4
  },
  "conversation_policy": {    // ★ 2층 — 시뮬 행동 지시로 직결
    "question_ratio": 0.35,          // 되묻기 성향
    "self_disclosure_pace": "느림",   // 사회침투 속도
    "reaction_amplitude": "큼",       // 리액션 크기 (헐/!! vs 담백)
    "conflict_mode": "지연-정리형",    // R-3에서: 즉시형|지연-정리형|회피형
    "reassurance_seeking": "낮음"     // 애착 축
  }
}
```

시뮬 프롬프트 v2 = 측정값 스타일 카드(1층) + conversation_policy 행동 지시(2층) +
"이 사람은 신경성 낮고 친화성 높음 — 부정 정서 단어 드묾, 완화 표현 자주" 트레이트 마커 지시(3층).

## 10. 정답지(response_preferences) — "받고 싶은 반응" (2026-07-04 신설, 백엔드 구현됨)

팀 제안 "유저에게 정답지를 받자(내가 어떤 질문을 하면 어떤 답변이 오는 게 좋은지)"의 설계.

- **심리학적 정체**: 개인별 '지각된 파트너 반응성(Reis — 이해·인정·관심)'의 명시적 정의.
  페르소나(나는 어떻게 말하나)와 직교하는 새 축: **나는 뭘 받고 싶나**.
- **철칙 1 — 시뮬 주입 금지**: 정답지가 시뮬 프롬프트에 들어가면 상대 에이전트가 정답지에
  맞춰 발화해 정직한 궁합 재현이 무너진다(모두가 모두와 잘 맞는 결과). 생성 단계가 아니라
  **평가(리포트) 단계 전용** 데이터다. 코드에서도 시뮬 페르소나 dict에 싣지 않는다.
- **철칙 2 — 매칭 하드필터 금지**: 명시적 선호는 실제 끌림을 예측하지 못한다
  (Eastwick&Finkel 2008 — 사전 이상형이 실제 데이트 끌림을 예측 실패; Joel et al. 2017 —
  100+ 특성에 ML을 걸어도 쌍 끌림 예측 불가). 리포트의 소프트 신호로만 쓴다.
- **구현(07-04)**: `personas.response_preferences` JSONB `[{code, situation, desired_reply,
  collected_at}]`. 수집 마커 `answer_letter="정답지"` — 페르소나 생성 프롬프트·말투 통계에서
  자동 제외(services/voice.py, prompts/persona.py). 리포트 프롬프트가 [받고 싶은 반응] 섹션으로
  양쪽 정답지를 받아 상대방AI의 실제 발화와 대조하고, 반응성(이해·인정·관심) 렌즈로 채점한다
  ("이것만으로 점수를 정하지 말 것" 지시 포함).
- **수집 형식 제안**: 데일리 주관식과 페어 — "상대가 '오늘 진짜 힘들었다ㅠ'라고 보냈어요.
  ① 당신은 뭐라고 답하나요?(=말투 표본) ② 당신이 힘들 때 어떤 답장을 받고 싶나요?(=정답지)".
  문항 하나로 voice와 preference 두 축이 모인다. 문항 확정은 팀 결정.
- **부수 효과**: 정답지 문장도 사용자가 직접 쓴 글 — 향후 습관 통계 표본으로 활용 검토 가능
  (단 '내가 보내는 말투'와 '받고 싶은 말투'는 다른 분포일 수 있어 v1에선 분리 유지).
