# AMORI Matching Module

페르소나 벡터 기반 매칭 알고리즘 + 시뮬레이션 오케스트레이션을 담당합니다. **명세현 담당.** 구현 방식은 자유롭게 선택하실 수 있습니다.

## 책임

- 누가 누구와 매칭되는지 결정 (페르소나 벡터 + 메타데이터 기반)
- 매칭된 쌍에 대해 시뮬레이션을 언제 / 어떻게 돌릴지 오케스트레이션
- 만남 후 피드백을 매칭 알고리즘에 반영하는 학습 루프

## 작업 시점

LLM (이현정) 과 백엔드 (손지민) 작업이 마무리된 이후 시작합니다. 그 전까지 백엔드는 단순 top-K 코사인 거리로 더미 매칭 결과를 반환하며, 명세현이 이를 본 모듈의 알고리즘으로 교체합니다.

## 만들어야 하는 것

### 1. 매칭 알고리즘

**입력**
- 쿼리 사용자의 페르소나 벡터 (1024차원)
- 후보 사용자 풀 (페르소나 벡터 + 메타데이터)
- 필터 (연령 등)

**출력**
- 후보 사용자 ID 와 매칭 점수의 랭킹된 리스트

베이스라인은 단순 코사인 유사도이지만, 가치관 / 유머 / 대화 패턴 등 카테고리별 가중치, 사용자 피드백 학습 등으로 발전시킬 여지가 큽니다.

### 2. 시뮬레이션 오케스트레이션

- 어떤 쌍을 시뮬레이션할지 결정 (사용자당 매일 N건 등)
- LLM 모듈의 `/llm/simulate` 를 호출하여 시뮬레이션 실행
- 결과(대화 로그 + 케미 점수)를 DB 에 저장
- 결과 알림은 백엔드의 푸시 시스템과 협조

### 3. 피드백 학습 루프 (제안서 차별점)

만남 후 사용자 피드백을 매칭 알고리즘에 반영합니다 (강화 루프).

## 통합 방식

백엔드와의 통합 방식은 자유롭게 결정합니다.

- HTTP API (백엔드와 별도 서비스)
- 라이브러리 import (백엔드 내 모듈)
- 워커 프로세스 (백그라운드 잡)

백엔드의 `/matches/find` 엔드포인트의 실제 매칭 로직과, 시뮬레이션 트리거가 본 모듈의 책임입니다.

## 작업 규칙

- 브랜치 prefix: `match/*`
- 작업 디렉토리: `matching/` 만 사용합니다. `backend/`, `llm/`, `lib/` 는 통합 PR 외에는 수정하지 않습니다.
- [`shared/schemas/`](../shared/schemas/) 변경 시 양쪽(이현정·손지민) 리뷰가 필수입니다.
